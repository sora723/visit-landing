# 접수 검증·허수 DB·IP 차단 가이드

> 대상: visit-landing 운영자 (현장관리 시트 · Apps Script @51+)  
> 코드: `78f8dcf` — `_검증로그` 선저장 → 성공 UI → 검수 → 알림 → (알림 성공만) 접수관리

---

## 0. 처리 흐름 (현재)

```
관심등록 submit
  → Master `_검증로그` 기록 (상태: 검수중 또는 조기차단)
  → 성공 UI («접수되었습니다»)
  → submit.postProcess (또는 구호환 notify.flush)
       → 검수(분류)
       → 알림톡 발송 시도
       → 알림 성공 건만 `접수관리` (+ 현장 미러)
```

- `_알림큐`에는 **신규 enqueue 하지 않음** (잔여 pending flush만)
- 광고경로(UTM/NaPm) 없어도 **광고신호없음**으로 알림톡 O · 전환 X (2026-07-23 실측)

---

## 1. 한눈에 보기

| 검증상태 | `_검증로그` | 알림톡 | 접수관리 | 네이버 전환 | 접수자 화면 | 용도 |
|----------|------------|--------|----------|------------|------------|------|
| **정상접수** | O | O | **알림 성공 시** | O | 완료 | 유효 DB |
| **빠른접수** | O | O | **알림 성공 시** | O* | 완료 | 빠르게 쓴 정상 후보 |
| **허수의심** | O | **O** | **알림 성공 시** | **X** | 완료 | 의심표시 — 담당자 확인 |
| **광고신호없음** | O | **O** | **알림 성공 시** | **X** | 완료 | 의심표시 — 광고 유입 아님 |
| **중복접수** | O | **O** | **알림 성공 시** | **X** | 완료 | 의심표시 — 24h 동일 번호 |
| **IP차단** | O | X | **X** | X | 완료 | 확정 차단 |
| **IP대량차단** | O | X | **X** | X | 완료 | 확정 차단 + `_IP차단` 24h |
| **허니팟차단** | O | X | **X** | X | 완료 | 확정 차단 |
| **토큰차단** | O | X | **X** | X | 완료 | 확정 차단 |

\* `ENABLE_FAST_SUBMIT_CONVERSION` 설정에 따름

**원칙**

- 접수자에게는 검증상태·차단 사유를 **절대 노출하지 않음** (항상 완료 화면)
- **허수의심만으로 자동 차단하지 않음** — 알림은 보냄, 네이버 전환만 막음
- **확정 차단**은 `_IP차단` 수동 · **IP대량차단** · 허니팟 · 토큰 없음
- **접수관리 = 알림톡 발송에 성공한 건** (알림 실패·확정차단은 `_검증로그`만)

## 2. 검증상태 설명

### 의심표시 (알림 O, 전환 X → 알림 성공 시 접수관리)

**허수의심** — 광고에서 왔어도 행동이 기계적일 때

- 제출까지 2초 이하, 입력 변경·포커스가 너무 적음 등
- 접수관리 `검증상태` = `허수의심`, `memo`에 사유 (`behavior`, `very_fast` 등)

**광고신호없음** — 네이버/UTM 유입 흔적이 없을 때

- NaPm, utm_source=naver, referrer naver 등 없음
- 직접 URL·북마크 접속 가능성
- **알림톡은 발송됨** (전환만 X)

**중복접수** — 같은 연락처로 24시간 내 이미 **정상·빠른** 접수가 있을 때

- 기존 접수 데이터는 삭제하지 않음

→ 담당자는 알림톡으로 확인 후, 진짜 허수면 `_IP차단`에 IP 추가

### 확정 차단 (접수관리 X, `_검증로그`만)

**IP차단** — 운영자가 `_IP차단` 시트에 IP를 넣었을 때

```
예: 115.22.75.135 GFA 허수작업 확인 후 차단
```

**IP대량차단** — 같은 현장·같은 IP로 짧은 시간에 너무 많이 접수할 때 (GFA 명의DB 묶음 공격 대응)

- 기본: **10분 안에 동일 IP 3건까지 허용**, **4건째부터** 차단
- 접수관리·알림톡·전환 없음 → `_검증로그`만 (`검증상태=IP대량차단`)
- **동시에 `_IP차단`에 24시간 임시 자동 등록** (`expiresAt` 설정, memo=`자동 24h IP대량차단 …`)
- 24시간 지나면 `expiresAt` 만료 → 해당 행은 더 이상 차단 안 됨 (수동 영구 차단은 `expiresAt` 비움)
- 확정 허수로 판단되면 운영자가 `_IP차단`에서 `expiresAt` 비우고 memo 수정 → **수동 영구 차단**

**허니팟차단** — 아래 「허니팟」 참고. 사람이 아닌 봇이 폼을 채울 때

**토큰차단** — 아래 「form_token」 참고. 페이지를 거치지 않고 API만 직접 호출할 때

---

## 3. 허니팟이란? (honeypot)

폼에 **사람 눈에는 안 보이는** 빈 칸 `company`를 숨겨 둡니다.

- 일반 고객: 보이지 않아 비움 → 정상 검증 진행
- 자동화 봇: HTML을 읽고 모든 input에 값을 넣음 → `company`에 값 있음 → **허니팟차단**

고객이 할 일 없음. 클릭·입력 추가 없음.

---

## 4. form_token (토큰차단)이란?

페이지를 열면 서버가 **10분짜리 일회용 토큰**을 발급합니다.

- 정상 흐름: 랜딩 로드 → 토큰 발급 → 폼 제출 시 토큰 포함 → 통과
- 비정상: Apps Script URL에 `submit`만 직접 POST (토큰 없음/만료) → **토큰차단**

인증번호 SMS가 아니라, **봇·스크립트 직접 호출 방지**용입니다.

---

## 5. IP 수동 차단 (`_IP차단` 시트)

Master 시트 → **`_IP차단`** 탭 (없으면 Apps Script 메뉴: VisitLanding → IP 차단 시트 추가)

| siteCode | ip | memo | enabled | createdAt | expiresAt |
|----------|-----|------|---------|-----------|-----------|
| L007 | 115.22.75.135 | GFA 허수 확인 (수동 영구) | Y | 2026-07-06 | |
| L008 | 211.234.56.78 | 자동 24h IP대량차단 … | Y | 2026-07-20 | 2026-07-21 |
| | 10.0.0.1 | 전 현장 차단 | Y | | |

- **siteCode** 비우면 → 모든 현장에 적용
- **enabled** = `Y` 일 때만 차단
- **expiresAt** 비우면 → 영구(수동). 값 있으면 해당 시각까지만 차단
- IP대량차단 4건째부터 → **자동**으로 24h 행 추가 (수동 영구 행은 덮어쓰지 않음)
- 차단 IP 접수 → 접수관리 **미저장**, `_검증로그`만, 알림톡·전환 없음

## 6. 시트 구조

### `_검증로그` (시도 원장)

- **모든 관심등록 시도** 1건당 1행 (차단 포함)
- submit 직후 `검수중` → postProcess 후 최종 `검증상태`로 갱신
- NaPm, utm, 행동 로그, IP, `raw_payload` 등 분석용 전체

### 접수관리 (알림 성공 리드)

- 기존 컬럼 + **`validationStatus` / `검증상태`**
- **알림톡 발송 성공 건만** 추가 (정상·의심 포함)
- `status`: 정상이면 `접수완료`, 의심이면 상태명
- `memo`: 의심 사유 (`no_ad_signal`, `behavior` …)

### `_IP차단`

- 수동·자동(24h) 차단 목록

### `_알림큐` (레거시)

- 신규 접수는 사용하지 않음. 예전 pending 잔여분 flush만

---

## 7. 네이버 전환

- **O**: `정상접수`, `빠른접수`
- **X**: 의심표시 전부, IP차단, IP대량차단, 허니팟, 토큰
- `/complete?verified=1` 일 때만 전환 스크립트 실행
- 참고: 성공 UI 응답 시점에는 `allowConversion=false` (검수는 postProcess). 전환은 후속 개선 여지

## 8. 설정 변경

`apps-script/SubmitValidationConfig.gs`

| 항목 | 기본값 | 설명 |
|------|--------|------|
| `FAST_SUBMIT_SECONDS` | 7 | 이상이면 정상 후보 |
| `VERY_FAST_SUBMIT_SECONDS` | 2 | 이하면 허수의심 요인 |
| `DUPLICATE_PHONE_TTL_SECONDS` | 86400 | 중복 판정 24h |
| `ENABLE_FAST_SUBMIT_CONVERSION` | true | 빠른접수도 전환할지 |
| `REQUIRE_AD_SIGNAL_FOR_CONVERSION` | true | 광고 신호 없으면 전환 X |
| `BULK_IP_WINDOW_SECONDS` | 600 | 동일 IP 대량 판정 시간 창(초) |
| `BULK_IP_MAX_COUNT` | 3 | 창 안 **3건 허용 · 4건째** `IP대량차단` |
| `BULK_IP_AUTO_BLOCK_HOURS` | 24 | IP대량차단 시 `_IP차단` 자동 등록 시간 |
| `VALIDATION_RECENT_LOG_ROWS` | 400 | IP 카운트 시트 폴백 시 최근 N행만 |
| `VALIDATION_RECENT_SUBMISSION_ROWS` | 800 | 전화 중복 검사 최근 N행만 |

IP 카운트는 `CacheService` 우선. submit은 `_검증로그`만 동기 기록하고, 검수·알림·접수관리는 `submit.postProcess`에서 처리.

변경 후 `npm run setup:apps-script:push` 로 배포.

---

## 9. 운영 절차 (허수 DB 발견 시)

1. `_검증로그` 또는 접수관리 `검증상태` 확인
2. 알림톡 온 건 전화·패턴 검증
3. **확정 허수** → `_IP차단`에 IP 추가
4. 네이버 광고는 의심 건 전환 학습 안 됨 (전환 X 정책)

---

## 10. 배포

```bash
cd visit-landing
npm run setup:apps-script:push   # Apps Script push + Web App 배포
git push origin main             # Netlify 프론트
```

Web App URL(`/exec`)은 배포 관리에서 **새 버전**이 반영되어야 `formToken.issue` 등이 동작합니다.
